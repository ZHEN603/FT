# 外贸聚合产品系统需求文档

版本：v1.0  
日期：2026-05-25  
阶段：MVP 内部使用，预留 SaaS 多租户扩展

## 1. 项目背景

外贸业务中，客户通常需要先浏览产品、选择规格、估算运输方式，再提交询价。业务人员随后需要在后台查看询价、管理客户、维护产品、供应商、加价规则，并持续跟进报价转化。

本系统定位为外贸聚合产品平台，前台面向海外客户展示产品并承接询价，后台面向内部业务团队管理产品、报价单、客户、供应商及跟进记录。MVP 阶段优先完成业务闭环，保证内部可以真实使用；后续再演进为可对外推广的 SaaS 多租户系统。

## 2. 项目目标

### 2.1 MVP 目标

- 支持内部团队维护产品、分类、规格、供应商、加价规则。
- 支持海外客户在前台浏览产品、选择规格、填写目的港并提交询价。
- 支持系统根据客户 IP 自动识别国家或地区，并允许客户手动修改。
- 支持后台自动接收前台询价，形成报价单记录。
- 支持业务人员维护客户资料、跟进记录和供应商资料。
- 支持基础仪表盘，帮助内部快速查看关键业务数据。
- 尝试实现产品采集、分析、报价换算、多语言、自动触达等差异化能力的最小可用版本。
- 技术实现以快速上线为优先，同时保留后续扩展空间。

### 2.2 SaaS 阶段目标

- 支持多租户，每个企业拥有独立数据、配置、品牌信息和前台站点。
- 支持更高并发、权限分层、审计日志、数据隔离和可观测性。
- 支持产品采集、数据分析、报价换算、汇率管理等增强能力。
- 支持企业级部署、计费、套餐、成员管理和租户级配置。

## 3. 用户角色

| 角色 | 使用端 | 主要目标 |
| --- | --- | --- |
| 海外客户 | 前台 | 浏览产品、选择规格、填写海运信息、提交询价 |
| 业务员 | 后台 | 查看报价单、维护客户、记录跟进、处理询价 |
| 产品运营 | 后台 | 维护产品分类、产品列表、规格、图片、加价规则 |

| 平台超级管理员 | 管理后台，二期 | 管理租户、套餐、平台配置、运营数据和异常处理 |

### 4.2 MVP 尝试实现或预留给二期

以下能力是产品差异化核心，不应被视为普通附加需求。MVP 阶段的策略是：优先打通最小闭环，复杂能力先采用半自动、人工确认、配置化或埋点预留的方式实现；当业务流程验证后，再在二期升级为自动化、规模化和可配置的 SaaS 能力。

- 产品采集
- 1688 源管理
- 分析管理：产品浏览分析、询盘分析、国家或地区分析、客户行为分析
- 报价换算
- 汇率管理自动化
- 多语言自动翻译
- 在线支付
- 客户前台账号体系
- 邮件、WhatsApp、CRM 深度集成

MVP 分层原则：

| 能力 | MVP 尝试版本 | 二期增强版本 |
| --- | --- | --- |
| 产品采集 | 支持导入已下载的网站离线包，解析 HTML/MHTML/Excel/JSON/图片资源并生成产品草稿 | 批量离线包导入、定时处理已下载文件、图片和规格自动清洗；长期探索全自动采集与注入 |
| 1688 源管理 | 保存源链接、源价格、供应商、下载批次、解析状态和原始文件引用 | 多源比价、供应商评分、价格变动提醒 |
| 分析管理 | 先做关键埋点和基础统计看板 | 完整行为分析、转化漏斗、客户画像、产品热度排行 |
| 报价换算 | 后台提供人工可编辑的报价计算器 | 自动计算产品价、加价、汇率、运费、税费和利润 |
| 汇率管理自动化 | 支持手动汇率和默认币种 | 汇率 API、历史汇率、生效时间和自动更新 |
| 多语言自动翻译 | 商品名称和描述支持人工维护多语言，预留自动翻译入口 | 接入翻译 API，批量翻译，人工审核发布 |
| 在线支付 | 先记录付款状态或外部付款链接 | Stripe、PayPal 等在线支付和订单闭环 |
| 客户前台账号体系 | MVP 可不强制注册，但预留客户识别和询价历史 | 客户登录、询价历史、收藏、订单和支付记录 |
| 邮件、WhatsApp、CRM 深度集成 | 先做邮件通知和 WhatsApp 跳转联系 | 自动邮件模板、WhatsApp API、CRM 双向同步 |

## 5. 核心业务流程

### 5.1 前台询价流程

1. 客户进入前台首页。
2. 客户浏览产品或进入全部产品列表。
3. 客户通过分类筛选产品。
4. 客户进入产品详情页。
5. 客户选择产品规格、数量等信息。
6. 客户进入集装箱或海运报价界面。
7. 系统根据 IP 自动识别客户国家或地区。
8. 客户填写或修改目的国家、目的港、运输相关信息。
9. 客户点击获取海运报价。
10. 系统展示海运报价或报价占位结果。
11. 客户点击提交询价。
12. 客户填写姓名、邮箱、电话、公司名、备注等联系方式。
13. 系统提交询价并生成后台报价单。
14. 后台报价单管理出现新询价记录。

### 5.2 后台处理询价流程

1. 业务员登录后台。
2. 业务员在报价单管理中查看新询价。
3. 业务员进入报价单详情查看客户、产品、规格、数量、目的港和备注。
4. 业务员可将客户信息沉淀到客户管理。
5. 业务员添加跟进记录。
6. 业务员根据供应商、加价规则和人工判断处理报价。
7. 报价单状态更新为已查看、跟进中、已报价、已成交或已关闭。

### 5.3 产品维护流程

1. 产品运营创建产品分类。
2. 产品运营创建产品并绑定分类。
3. 产品运营维护产品图片、名称、描述、规格、价格、供应商等信息。
4. 产品运营设置产品状态为上架或下架。
5. 前台仅展示上架产品。
6. 当价格策略变化时，产品运营维护加价规则。

### 5.4 离线产品注入流程

为降低实时爬虫风险，MVP 的产品上架注入不采用系统直接爬取第三方网站的方式，而采用“人工下载网站内容 -> 系统离线解析 -> 人工确认入库”的半自动流程。

1. 运营人员在外部环境中人工下载目标网站内容。
2. 下载内容形成离线导出包，包含 HTML/MHTML、Excel、JSON、图片资源等文件。
3. 运营人员在后台上传离线导出包，或将导出包放入系统指定目录。
4. 系统读取导出包并生成导入批次。
5. 系统解析商品分类、商品标题、价格、主图、SKU、颜色、源链接、产品详情和图片资源。
6. 系统根据去重规则合并重复商品或重复图片。
7. 系统生成产品草稿和解析报告。
8. 运营人员检查草稿内容、图片、分类、价格、规格和供应商来源。
9. 运营人员确认后，将产品草稿批量或单个注入正式产品库。
10. 正式产品保留来源文件、源链接、导入批次和解析日志，方便后续追溯。

开发样例：

- 当前仓库 `misc/export` 目录包含一份已下载并整理的样例导出包。
- 样例包含 `catalog_data.json`、SKU 明细 Excel、HTML 商城文件、原始图片目录和优化图片目录。
- 开发产品导入能力时，可优先使用该样例包作为解析、字段映射、图片入库和去重逻辑的测试数据。

![后台登录界面](assets/admin-login.png)

- 后台用户通过账号和密码登录。

### 7.5 产品采集与 1688 源管理

功能说明：

- 产品采集和 1688 源管理是提升选品和上架效率的关键差异化能力。
- 为避免实时爬虫风险，MVP 必须采用离线导入方案：先由人工下载目标网站内容，再由系统解析离线包并生成产品草稿。
- MVP 不要求系统直接访问第三方网站，也不要求全自动采集；重点是完成“离线包导入、内容解析、去重清洗、草稿生成、人工确认入库”的闭环。
- 产品长期目标仍然是尽可能自动化地完成“来源发现、内容获取、解析清洗、产品注入、持续更新”，但是否能做到全自动，需要在技术可行性、合规风险、平台规则、稳定性和成本上持续探索。

MVP 必做版本：

- 支持上传或选择离线导出包。
- 支持识别导出包中的 HTML、MHTML、Excel、JSON、图片目录等资源。
- 支持解析商品分类、商品标题、外贸批发价、主图、颜色 SKU 图组、可选颜色、1688 链接和产品详情。
- 支持读取解析报告，包括分类汇总、源页面数量、去重数量、异常状态和失败原因。
- 支持按分类、SKU 主体、颜色、标题、图片 hash 等规则进行去重。
- 支持将采集结果保存为草稿产品。
- 支持人工编辑草稿后发布为正式产品。
- 支持正式产品追溯来源链接、来源文件、导入批次和解析日志。
- 支持导入状态管理：待解析、解析中、已生成草稿、部分失败、已入库、已忽略。

样例数据：

| 文件或目录 | 说明 |
| --- | --- |
| `misc/export/catalog_data.json` | 分类汇总、去重规则、源页面状态和解析元数据 |
| `misc/export/1688衣架SKU明细_无待核无错图去重版.xlsx` | SKU 明细表，可用于字段映射和批量导入测试 |
| `misc/export/1688衣架独立站商城_无待核无错图去重版.html` | 已生成的静态商城 HTML，可用于前台展示结构参考 |
| `misc/export/assets/` | 原始图片资源目录 |
| `misc/export/assets_optimized/` | 优化后的图片资源目录 |

字段要求：

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| 导入批次号 | 是 | 系统生成，用于追溯本次离线导入 |
| 导入包名称 | 是 | 上传文件名或目录名 |
| 导入包路径 | 否 | 服务器内部路径或对象存储路径 |
| 源平台 | 是 | MVP 默认 1688 或手动选择 |
| 源文件类型 | 是 | HTML、MHTML、Excel、JSON、图片目录等 |
| 源链接 | 否 | 1688 商品链接或来源页面链接 |
| 源商品名称 | 否 | 从离线包解析或人工录入 |
| 源价格 | 否 | 从离线包解析，用于报价参考 |
| 源图片 | 否 | 本地图片路径、对象存储路径或图片 URL |
| 源规格 | 否 | 规格文本或结构化 SKU |
| 关联供应商 | 否 | 可选择已有供应商或创建供应商 |
| 去重规则 | 否 | 如同分类、同标题、同 SKU 主体、同颜色、同图片 hash 去重 |
| 解析状态 | 是 | 待解析、解析中、已生成草稿、部分失败、已入库、已忽略 |
| 解析报告 | 否 | 分类数量、成功数量、失败数量、异常说明 |
| 失败原因 | 否 | 解析失败或入库失败时记录 |

阶段路线：

| 阶段 | 目标 | 说明 |
| --- | --- | --- |
| MVP | 半自动离线导入 | 人工下载目标网站内容，系统解析离线包并生成产品草稿 |
| 二期 | 自动化处理已下载内容 | 支持批量离线包、定时扫描指定目录、自动清洗图片和规格 |
| 长期探索 | 全自动采集与注入 | 在合规和技术条件允许时，探索自动发现来源、自动获取内容、自动更新产品 |

二期增强：

- 支持更丰富的离线包格式。
- 支持批量上传多个离线包。
- 支持图片自动入库和压缩优化。
- 支持规格、价格、库存结构化清洗。
- 支持导入记录管理。
- 支持草稿批量编辑后入库。
- 支持源链接、源价格、来源文件、供应商信息留存。
- 支持价格变化提醒和源商品失效提醒。
- 在合规和风控允许时，评估自动采集、接口化采集或授权数据源接入。

长期探索：

- 自动发现或接入商品来源。
- 自动获取商品页面、SKU、图片、价格、供应商等数据。
- 自动识别页面结构变化并调整解析策略。
- 自动生成产品草稿并标注置信度。
- 自动更新已入库产品的价格、图片、规格和供应商状态。
- 对高风险来源启用人工审核和风控拦截。
- 是否进入正式开发取决于合规评估、技术稳定性、数据质量和投入产出比。

验收标准：

- 后台可以上传或选择离线导出包。
- 系统可以解析样例包 `misc/export` 中的 JSON、Excel、HTML 和图片资源。
- 系统可以生成导入批次和解析报告。
- 系统可以根据解析结果生成产品草稿。
- 产品草稿经人工编辑后可以入库为正式产品。
- 正式产品可追溯来源链接、来源文件、导入批次和供应商。
- 即使部分文件解析失败，也允许人工补充信息继续入库。

### 7.11 分析管理

功能说明：

- 分析管理用于判断产品热度、询价转化、客户来源和国家或地区分布，是后续运营和 SaaS 价值展示的关键模块。
- MVP 应优先完成事件埋点和基础统计，不要求一开始实现完整 BI。

MVP 埋点事件：

- 产品浏览事件
- 分类点击事件
- 提交询价事件
- 获取海运报价事件
- 国家或地区识别结果
- 搜索关键词事件
- 产品规格选择事件
- 询价提交失败事件

MVP 基础看板：

| 看板 | 指标 |
| --- | --- |
| 产品浏览分析 | 浏览量、访客数、热门产品 |
| 询盘分析 | 询价总数、今日新增、转化产品排行 |
| 国家或地区分析 | 访客国家或地区、询价国家或地区 |
| 客户行为分析 | 浏览产品数、提交询价路径、最近访问时间 |

数据采集要求：

- 前台关键动作需记录事件名称、时间、产品、分类、会话标识、IP 国家或地区、来源页面。
- MVP 可使用匿名 session 标识客户行为，不强制客户登录。
- 若客户提交询价，应将匿名行为与客户和报价单关联。

二期增强：

- 转化漏斗分析
- 产品热度排行
- 客户画像
- 来源渠道分析
- 租户维度分析
- 自定义报表

验收标准：

- 产品详情页访问可以形成浏览记录。
- 提交询价可以形成询盘统计。
- 后台能查看基础产品浏览和询价统计。
- 同一客户提交询价后，可在客户详情中看到其关键行为摘要。

### 7.12 报价换算

功能说明：

- 报价换算用于帮助业务员快速从产品成本、加价规则、汇率、运费等信息生成参考报价。
- MVP 可先做后台内部使用的报价计算器，不直接向客户承诺最终价格。

MVP 尝试版本：

- 支持产品价格、加价、汇率、运费、税费等综合换算。
- 支持不同币种报价。
- 支持根据产品基础价格和加价规则计算参考价。
- 支持业务员手动输入运费、杂费、折扣、利润率。
- 支持计算结果保存到报价单详情。
- 支持人工修改最终报价。

字段要求：

| 字段 | 必填 | 说明 |

| 加价规则 | 否 | 可自动匹配或手动选择 |
| 币种 | 是 | 默认 USD |

| 利润率 | 否 | 用于参考报价 |
| 最终报价 | 是 | 业务员确认后的报价 |

二期增强：

- 自动匹配汇率、运费、税费和加价规则。
- 支持报价单导出 PDF。
- 支持邮件发送报价。
- 支持报价版本管理。
- 支持客户在线确认报价。

验收标准：

- 业务员可以在报价单详情中进行报价换算。
- 计算结果可保存为报价单的参考价或最终报价。
- 业务员可覆盖自动计算结果。

### 7.14 多语言与自动翻译

## 11. 非功能需求

### 11.1 性能

MVP：

- 后台列表页响应时间建议小于 1 秒。
- 前台页面首屏尽量控制在 3 秒内。
- 产品列表、报价单列表、客户列表必须分页。

SaaS：

- 支持租户级横向扩展。
- 支持缓存热点数据。
- 支持队列处理邮件、采集、分析等异步任务。

### 11.2 安全

- 后台接口必须鉴权。
- 密码必须加密存储。
- 表单需做前后端双重校验。
- 防止 SQL 注入、XSS、CSRF 等常见风险。
- 图片上传需限制格式和大小。
- SaaS 阶段需强化租户数据隔离。

### 11.3 可维护性

- 前后台业务模块边界清晰。
- 数据模型需为多租户预留 tenant_id 或可迁移方案。
- 前台展示与后台管理解耦。
- 报价单使用快照字段，避免历史数据被关联数据变更污染。

### 11.4 兼容性

- 前台适配桌面端和移动端。
- 后台优先适配桌面端，平板可用，移动端可不重点优化。
- 支持主流现代浏览器。

## 12. 技术选型建议

### 12.1 MVP 技术选型原则

- 最快速上线。
- 包含完整业务闭环。
- 开发和部署成本低。
- 代码结构为后续扩展和迁移保留空间。
- 避免过早引入复杂微服务。

### 12.2 MVP 推荐方案

推荐方案：单体全栈应用。

| 层级 | 建议 |
| --- | --- |
| 前端和后台 | Next.js 或类似全栈框架 |
| UI | Ant Design、shadcn/ui 或成熟后台模板 |
| 数据库 | PostgreSQL |
| ORM | Prisma 或 Drizzle |
| 鉴权 | Auth.js、Lucia 或自研轻量 session |
| 文件存储 | 本地对象存储兼容方案，或直接使用 S3/R2/OSS |
| 部署 | VPS、Docker Compose、Vercel + 托管数据库均可 |
| IP 识别 | 第三方 GeoIP API 或本地 GeoIP 库 |

说明：

- 如果团队前端能力较强，Next.js 可以同时承载前台、后台和 API，最快完成 MVP。
- 如果后台复杂度明显增长，可后续拆出独立 API 服务。
- 数据库优先使用 PostgreSQL，方便后续做多租户、搜索、报表和事务处理。

### 12.3 SaaS 企业级阶段建议

| 层级 | 建议 |
| --- | --- |
| 前台 | Next.js，支持租户域名和 SEO |
| 后台 | Next.js 或独立 React Admin |
| API | NestJS、Fastify、Go 或 Java Spring Boot |
| 数据库 | PostgreSQL，按 tenant_id 数据隔离，必要时演进到独立库 |
| 缓存 | Redis |
| 队列 | BullMQ、RabbitMQ、SQS 等 |
| 搜索 | PostgreSQL Full Text、Meilisearch 或 Elasticsearch |
| 文件存储 | S3、Cloudflare R2、阿里云 OSS |
| 监控 | Sentry、OpenTelemetry、Prometheus、Grafana |
| 部署 | Docker、Kubernetes 或云平台托管服务 |

多租户策略建议：

- 初期 SaaS：共享数据库、共享表、tenant_id 隔离。
- 中期增长：大租户可迁移到独立 schema 或独立数据库。
- 所有业务表从设计上预留 tenant_id，MVP 即使单租户也建议保留或确保后续可平滑补充。

## 15. 风险与决策点

| 风险或决策点 | 建议 |
| --- | --- |
| 海运报价是否接真实 API | MVP 可先做人工报价或占位结果，避免对接成本拖慢上线 |
| 实时爬虫风险 | MVP 不做系统实时爬取，采用人工下载离线包后再导入解析 |
| 离线包格式不稳定 | MVP 先支持样例包中的 JSON、Excel、HTML 和图片目录，再逐步扩展格式 |
| 产品采集复杂度 | MVP 先做离线包解析、草稿和人工确认；二期提升离线处理自动化；长期探索全自动采集与注入 |
| 全自动采集合规不确定性 | 全自动是长期目标，不作为 MVP 承诺；进入开发前需评估目标网站规则、授权方式、访问频率、数据使用边界和法律风险 |
| 全自动采集技术稳定性 | 需验证页面结构变化、反自动化策略、图片质量、SKU 解析准确率和异常恢复能力 |
| 分析管理数据不足 | MVP 先埋点并做基础统计，避免一开始建设复杂 BI |
| 自动翻译准确性 | MVP 先允许人工维护英文内容，自动翻译必须人工审核后发布 |
| 在线支付合规和回调复杂度 | MVP 先记录付款状态和外部付款信息，真实支付二期接入 |
| 是否第一版就做多租户 | MVP 内部使用不建议完整实现，但建议数据模型预留 |
| 是否做客户账号体系 | MVP 不强制注册，但保留客户身份识别和询价历史的数据基础 |
| 报价是否自动计算 | MVP 可做参考价和人工确认，不强制自动出正式报价 |
| WhatsApp 和 CRM 接口限制 | MVP 先做快捷跳转和字段预留，API 集成二期推进 |
| 后台权限复杂度 | MVP 可简化为管理员，预留角色字段 |
